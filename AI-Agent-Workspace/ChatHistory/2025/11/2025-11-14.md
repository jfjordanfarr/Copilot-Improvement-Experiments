GitHub Copilot: - **Python Docstring Families**  
  - **PEP 257 / Plain reST**: freeform prose plus reStructuredText field lists. Sphinx treats `:param name:`, `:type name:`, `:keyword name:`, `:kwtype name:`, `:raises:`/`:except:`, `:var name:`, `:vartype name:`, `:ivar:`, `:cvar:`, `:returns:`, `:rtype:`, `:yield:`, `:ytype:`, `:deprecated:` (with optional version/reason), `:since:`, `:note:`, `:warning:`, `:attention:`, `:todo:`, `:seealso:`, `:example:`/`:examples:`, `:permission:`, `:requires:`, `:versionadded:`, `:versionchanged:`, `:seealso:`, etc. Any custom role introduced via `rst_prolog` can appear, but these are the stock ones the Sphinx domain understands and can map directly to sections in a Live Doc.  
  - **Google/Napoleon style**: section headers are literal words that Napoleon normalizes. Canonical headings include `Args`, `Arguments`, `Parameters`, `Params`, `Kwargs`, `Keyword Args`, `Keyword Arguments`, `Returns`, `Return`, `Raises`, `Exceptions`, `Yields`, `Yield`, `Attributes`, `Class Attributes`, `Instance Attributes`, `Warns`, `Warnings`, `See Also`, `Notes`, `Other Parameters`, `References`, `Examples`, `TODO`, `Deprecated`, `Requires`, `Since`. Rows under these sections follow `name (type): description` or bullet syntax; Napoleon also supports Google-style in-function attribute docs (`Attributes:` section inside classes).  
  - **NumPy/Napoleon style**: defined blocks: `Parameters`, `Other Parameters`, `Returns`, `Yields`, `Receives`, `Raises`, `Warns`, `Warnings`, `See Also`, `Notes`, `References`, `Examples`, `Attributes`, `Methods`. Each block uses table-like indentation: `name : type` lines followed by indented descriptions, optionally multiple outputs. Napoleon converts these into Sphinx nodes.  
  - **Epytext / Javadoc-like**: less common today but still around (`@param name`, `@type`, `@rtype`, `@return`, `@raise`, `@ivar`, `@cvar`, `@var`, `@group`, `@undocumented`). Twisted’s tooling and PyDoctor understand these tags; Sphinx can via extensions.  
  - **Doctest Blocks**: triple-quoted sections with `>>>` prompts; good to preserve for Live Doc `Examples`. These often sit under `Examples`/`Example` headings or follow `.. doctest::`.  
  - **Directive Blocks**: embedded `.. note::`, `.. warning::`, `.. admonition::`, `.. code-block:: python`, `.. rubric::`, etc. We can inline them as structured remarks if we parse the reST AST or conservatively treat them as prose.

  - **Detection**: keep the current triple-quote scan, but run the body through a lightweight classifier that distinguishes Google vs NumPy vs field-list vs epytext.   - **Extraction Targets**  
    - Summary: first non-empty paragraph.  
    - Parameters / Keyword Parameters: `:param`, `Args`, `Parameters`, `Other Parameters`.  
    - Types: `:type`, `name : type`, `(type)` parenthetical.  
    - Returns / Yields: `:returns:`, `Returns`, `Yields`.

    - Attributes: `Attributes`, `:ivar:`, `:var:`.
