jfjordanfarr: I want to talk first about a kind of pivot and a kind of not-pivot: more of a grand simplification. 

Yesterday, we worked a refactor in `fallbackInference.ts` because **we found a simpler way to do what we wanted**.

Today, I want to talk at a much grander scale, at the Vision layer itself, about a simplification which still nets us all of what we have now, but for a hell of a lot less work. 

Over the course of development, we've been building up tooling which:
- Enforces lint-like rules on markdown documents following certain glob patterns
- Bridges between documentation files and source code files
- Builds workspace-wide polyglot AST.
- Writes comments to source code files to indicate a pairing with a piece of "Live Documentation" (we use that phrasing to avoid accidentally enforcing via our tooling's default settings our lovely-for-us-but-totally-unknown-anywhere-else convention of MDMD)
- (Not yet built) bridges docstrings between public symbols/programmatic surface on the "live documentation" and source code file(s) themselves.

But... I want to talk about a remarkable feature of Markdown links which made MDMD possible in the first place: **Markdown Links _are_ their own form of AST**. 

I want to provide, now, a somewhat simpler vision of _how_ we build workspace-wide AST-like intelligence: We create a "Live Documentation" file (to us, that's a Layer 4 MDMD doc) for **every single file asset for which we want AST-capable visibility**.
For each "Live Documentation" file, we must programmatically extract from the source code file:
- Public symbols and any docstring-like feature on them (including the ability to correctly docstrings from things like `/// <remarks/>` and other derived doc-like indicators, translating them into markdown links to the original docstring/remarks, and possibly even later on down the line we could provide the ability to use shebang-like syntax to preview the original docstring/remarks from the markdown link in the derived class's corresponding Live Documentation file)
- Every **direct** interacting file (parent class file, interface file, files that this class speaks to directly and knows about), and, if possible, the public symbol(s) from those files for best granular AST help and workspace-consequence predictions. For interactors which are non-plaintext files (i.e. when we need to document a link between an HTML file and an image file that it displays), smaller "Live Documentation" files are likely permissible, just to provide us the AST window into their existence at the MarkDown level.


The overall vision: A VS Code extension called "Live Documentation" which:
- Creates a "Live Documentation" file for each file matching glob pattern(s), output to a workspace directory configurable by the user but defaulting to `/.live-documentation/`, giving the user the simple ability to version or not version the docs within their git repository with ease
- Mirrors the directory structure of the source files for which "Live Documentation" files are created (maximize similarity to source code paths to reduce LLM hallucinations)

...where a "Live Documentation" file contains a relatively constant sized **Authored portion** at the variable-sized **Generated portion**. 
The **Authored portion** would always come first in these documentation files to permit abritrarily long generation of markdown files for code files which contain huge numbers of symbols, without causing LLMs to need to read thousands of files to locate the actual "authored" portions that they want to edit of a piece of "Live Documentation". The **Authored portion** would contain these kinds of sections:
- **Description**: What does this file do?
- **Purpose** (something that we talk about a lot in our instructions files -- we want all of our code to justify its existence; if it can't, that's a signal of technical debt, project bloat, or old AI slop files): Why does this file exist? What happens if this file gets modified or deleted? What requirement or architectural constraint mandated the existence of this file? What is this file providing to the overall system?
- **Notes**: Any important information we want to provide our future selves with respect to this file (i.e. to help prevent us from making repeat mistakes, or to get the jump on any potential common misunsderstandings that we've seen more than once during our work). Sticky notes and miscellany that is genuinely useful (we do not like excessive documentation or excessive code) but cannot be reliably measured by the **Generated portion**.
